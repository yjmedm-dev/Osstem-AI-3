# 개발 초보자를 위한 프로젝트 설명서

> "개발을 전혀 모른다"는 전제로 쉽게 설명합니다.

---

## 이 프로그램은 무엇을 하는 건가요?

간단히 말하면:

> **해외 법인에서 보내온 엑셀 재무 파일을 받아서,
> 잘못된 숫자나 이상한 항목이 있는지 자동으로 검사하고,
> AI가 분석해서 결과 보고서를 만들어 주는 프로그램입니다.**

지금까지 담당자가 눈으로 직접 확인하던 일을
컴퓨터가 대신 자동으로 해줍니다.

---

## 프로그램이 일하는 순서 (요리 비유)

```
① 재료 받기     →  해외 법인이 보낸 엑셀 파일을 받음
② 재료 손질     →  파일 안의 숫자를 컴퓨터가 읽기 좋게 정리
③ 품질 검사     →  숫자가 맞는지, 빠진 항목이 없는지 검사
④ 전문가 의견   →  AI(Claude)가 이상한 점이 있는지 분석
⑤ 보고서 작성  →  결과를 엑셀 또는 PDF로 출력
```

## 폴더 구조 — "서랍장"에 비유

프로젝트 폴더는 서랍장이라고 생각하세요.
각 서랍마다 담당하는 역할이 다릅니다.

```
Osstem-AI-3/               ← 서랍장 전체
│
├── main.py                ← [시작 버튼] 프로그램을 켜는 스위치
│
├── config/                ← [설정 서랍] 비밀번호·주소 같은 설정 보관
│   └── settings.py            API 키, 데이터베이스 주소 등
│
├── models/                ← [양식 서랍] 데이터의 모양(틀)을 정의
│   ├── subsidiary.py          법인 정보 양식 (이름, 국가, 통화 등)
│   ├── account.py             계정과목 양식 (코드, 이름, 유형 등)
│   └── trial_balance.py       시산표 양식 (금액, 기간 등)
│
├── ingestion/             ← [입력 서랍] 파일을 읽고 정리
│   ├── excel_parser.py        엑셀 파일을 열어서 내용을 읽는 도구
│   ├── schema_mapper.py       법인마다 다른 계정코드를 표준으로 맞추는 도구
│   └── data_normalizer.py     통화 단위, 날짜 형식을 통일하는 도구
│
├── validation/            ← [검사 서랍] 숫자가 맞는지 검사
│   ├── engine.py              모든 검사를 순서대로 실행하는 총괄 담당
│   └── rules/                 검사 항목들 (규칙 모음)
│       ├── arithmetic_rules.py    덧셈이 맞는지 확인
│       ├── period_rules.py        작년이랑 비교해서 너무 차이나는지 확인
│       └── anomaly_rules.py       이상한 숫자가 있는지 확인
│
├── analysis/              ← [AI 서랍] Claude AI에게 분석 요청
│   ├── claude_client.py       AI와 대화하는 통로
│   └── prompt_builder.py      AI에게 보낼 질문을 만드는 도구
│
├── reporting/             ← [출력 서랍] 결과물 만들기
│   ├── excel_reporter.py      결과를 엑셀로 저장
│   └── pdf_reporter.py        결과를 PDF로 저장
│
└── utils/                 ← [도구 서랍] 여러 곳에서 공통으로 쓰는 도구들
    └── currency.py            환율 계산 도구
```

## 핵심 개념 설명

### 1. `main.py` — 시작점
레고 조립의 "설명서 첫 페이지"입니다.
프로그램을 실행하면 가장 먼저 이 파일이 실행되고,
나머지 파일들을 순서대로 불러서 일을 시킵니다.

```python
# 사용 예시 (터미널에서 실행)
python main.py --corp US001 --period 2024-12 --file tb.xlsx
#              ↑ 법인코드     ↑ 결산기간        ↑ 엑셀파일명
```

---

### 2. `models/` — 데이터 모양(틀) 정의

데이터베이스에 정보를 저장하기 전에,
어떤 모양으로 저장할지 "양식지"를 먼저 만드는 것입니다.

예를 들어 `subsidiary.py` (법인 양식)는 이런 정보를 담습니다:

| 항목 | 예시 |
|------|------|
| 법인 코드 | US001 |
| 법인명 | Osstem America Inc. |
| 국가 | United States |
| 통화 | USD |
| 회계기준 | IFRS |
| 결산월 | 12월 |

---

### 3. `ingestion/` — 파일 읽기 & 정리

**비유:** 해외에서 보낸 소포를 뜯어서 내용물을 꺼내 정리하는 과정

- `excel_parser.py` → 엑셀 파일을 열어서 숫자와 글자를 읽음
- `schema_mapper.py` → 법인마다 계정코드가 다를 수 있으므로, 우리 회사 기준으로 맞춰줌
  - 예: 미국 법인은 "1010"이 현금이고, 베트남 법인은 "111"이 현금일 때 → 둘 다 "현금"으로 통일
- `data_normalizer.py` → 통화(USD→KRW), 날짜 형식(2024-12-31 vs 31/12/2024) 통일

---

### 4. `validation/` — 검증 엔진 (가장 중요!)

**비유:** 세무조사관이 체크리스트를 들고 숫자를 하나씩 확인하는 과정

#### 검사 항목 (규칙) 종류:

**AR — 산술 검증 (Arithmetic Rules)**
> 기본 덧셈이 맞는지 확인

| 규칙 | 설명 | 심각도 |
|------|------|--------|
| AR-001 | 차변 합계 = 대변 합계인지 확인 | 🔴 CRITICAL (제출 불가) |
| AR-002 | 자산 = 부채 + 자본 인지 확인 | 🔴 CRITICAL |
| AR-003 | 이익잉여금이 맞게 계산됐는지 확인 | 🟠 ERROR (수정 필요) |

**PR — 기간 비교 검증 (Period Rules)**
> 작년이랑 비교해서 너무 많이 바뀐 항목이 없는지 확인

| 규칙 | 설명 | 심각도 |
|------|------|--------|
| PR-001 | 전기 대비 30% 이상 변동 항목 탐지 | 🟡 WARNING (주의) |

**AN — 이상값 탐지 (Anomaly Rules)**
> 상식적으로 말이 안 되는 숫자를 찾아내는 검사

| 규칙 | 설명 | 심각도 |
|------|------|--------|
| AN-001 | 필수 계정의 잔액이 0원인 경우 | 🟡 WARNING |
| AN-002 | 자산인데 마이너스(-)인 경우 | 🟠 ERROR |
| AN-003 | 수상하게 딱 떨어지는 숫자가 너무 많은 경우 | 🟡 WARNING |

#### 심각도 기준:

```
🔴 CRITICAL  → 이 오류가 있으면 제출 자체가 불가능
🟠 ERROR     → 반드시 수정 후 다시 제출
🟡 WARNING   → 확인은 해야 하지만 제출은 가능
🔵 INFO      → 참고 정보 (조치 불필요)
```

---

### 5. `analysis/` — AI 분석

검증이 끝난 결과를 Claude AI에게 보내서
"이 재무제표에서 주의해야 할 점이 뭔지" 물어봅니다.

**비유:** 숫자 확인은 직원이 했고, 최종 판단은 베테랑 회계사에게 맡기는 것

```
우리 프로그램이 Claude에게 보내는 질문 예시:

"US001 법인의 2024년 12월 결산 자료입니다.
 - CRITICAL 오류 2건 발견
 - 매출액이 전기 대비 45% 감소
 - 미수금이 전기 대비 80% 증가

이 내용을 분석하고 원인과 조치 방안을 알려주세요."
```

---

### 6. `reporting/` — 결과 보고서 만들기

모든 검증과 분석이 끝나면 결과를 정리해서 보고서로 만듭니다.

- **엑셀 보고서**: 오류 항목 목록, 법인별 검증 결과 요약
- **PDF 보고서**: 경영진 보고용 깔끔한 요약본

## 개발 순서 (로드맵)

처음부터 모든 기능을 만들 필요는 없습니다.
단계별로 하나씩 만들어 갑니다.

```
1단계 (기반)     폴더 만들고, 데이터 구조 설계
    ↓
2단계 (읽기)     엑셀 파일을 읽어서 저장하는 기능
    ↓
3단계 (검증)     숫자 오류를 찾아내는 기능 ← 핵심!
    ↓
4단계 (AI)       Claude AI 분석 연동
    ↓
5단계 (출력)     결과 보고서 자동 생성
    ↓
6단계 (고도화)   환율 자동화, 웹 화면, ERP 연동 등
```

## 자주 쓰는 개발 용어 설명

| 용어 | 쉬운 설명 |
|------|---------|
| API | 두 프로그램이 대화하는 방식. Claude API = Claude와 대화하는 통로 |
| 파싱(Parsing) | 파일이나 텍스트에서 필요한 정보만 뽑아내는 것 |
| 모델(Model) | 데이터를 저장할 때의 틀(양식) |
| ORM | 데이터베이스를 파이썬 코드로 다루는 방식 |
| 마이그레이션 | 데이터베이스 구조를 변경할 때 쓰는 도구 |
| 환경변수(.env) | API 키처럼 외부에 노출되면 안 되는 설정값 보관 파일 |
| 테스트 코드 | 내가 만든 코드가 제대로 작동하는지 자동으로 확인하는 코드 |
| CLI | 마우스 없이 키보드 명령어로 프로그램을 실행하는 방식 |
